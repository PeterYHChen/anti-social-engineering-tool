Introduction

As technologies advance, more and more people are relying on the internet or cloud platforms to store and access their personal information as well as assets. To protect customers’ assets, companies that provide information services invest a considerable amount of money into building a secured ease-of-use platform. In consequence, it has become harder and harder for malicious parties to attack or steal valuable customer information from the bigger companies. Therefore, these malicious hackers are shifting their focus more toward the customer side, emphasizing greater social engineering tactics to psychologically manipulate people to perform actions that lead to exposure their confidential information.

The social engineering attack is all about the weakness of humans. There is no single software, application or firewall to block such attacks.

For small companies and startups, it seems too expensive or even unaffordable to establish a security team to avoid this kind of attack. For a single user of Internet, it is impossible to detect all kinds of social engineering attacks.

Unfortunately, there is no guidebook to elaborate all kinds social engineering attacks. One possible reason is that attack methods are changing really quickly. Therefore, we come up with this idea to compose such a guidebook to help people, startups and organizations to protect their privacy.

Password Security

Password security is the most important issue in privacy security. Since almost everyone uses password everyday, the risk of password disclosure can be extremely high. Even though some scientists came up with the idea that we can use other biometrics as password such as Touch-ID or Face-ID, we think that the traditional password is the most reliable verification method (When you reboot your iPhone, the security system requires the cipher-based password instead of biometric password).

In this chapter, we provide the recommendations and industrial best practices in password security. According to different needs for general users, software engineers and product manager, we divide this chapter into three parts.

One additional way to protect user’s account is using multi-factor authentication. Multi-factor authentication is the second lock to make sure the security of the account when the password disclosed. For a general user, using multi-factor authentication can lower the risk of unwilling logins.

Software Engineers

For software engineers, password security is not only about the cipher-based password itself, but also how to implement a secure system to store, transmit and certification password. Protecting user’s password is also important. In anti-social engineering field, software engineers have to take actions when user’s password have been stolen in other applications.

Software engineers should use an appropriate way to store user’s password. In best practice, an application should store the hashed password in its database. Some software engineers make mistakes when hashing and storing password and it becomes chaos when attacks come.

1. Hashing password

Storing the raw password of users is irresponsible, always store the hashed password. The hash function should be asymmetric and should not be easily cracked. Adding salt promises that when the hashed password leaked, it would not become a disaster for users and your company.

2. Password protection

In anti-social engineering field, password protection is about to protect the database password and user’s password and even magic key (we don’t recommend software engineers use magic key, but they actually did) to avoid password leakage by source code leakage or fraud.

Do not provide password certification APIs to other application. Some social engineering attackers forge non-existent web application and require software engineers to provide password certification APIs for them. Providing such APIs to attackers helps them to achieve account credential enumeration attack. One of the most influential social engineering attacks about password certification APIs happened in 2015, when a group of swindlers required password certification APIs of a famous Chinese social networking company for their non-existent application, and by analyzing the APIs, they successfully found the way to bypass the authentication.

Another threat about password in social engineering field is the magic key. The magic key is the master key for software engineers, customer service team or data analysts, to get the easy access to the database, systems, background interface, etc..  Even though magic key is extremely irresponsible, some start-ups choose to set a magic key to cut cost in employee training or company management. As a software engineer, if you have to use magic key, do not store it in the source code, store it in the database instead. One of the serious consequences of storing magic key in the source code is if you don’t change it by re-compiling (re-compiling is costly and most companies won’t do that for security reason) your code when an employee leave, the magic key leaks.

The most important thing in password protection is protection the engineers’ passwords. However, many engineers don’t protect their passwords in a professional way. In 2016, a outer attacker used social engineering attack to lure four Mobi software engineers to disclose their passwords and got the access to their GitHub account. Therefore, we believe the password security for software engineers are more severe than which of general users.

Product Managers

When designing the application, product managers should take password security into consideration. The designing phase of the product should not only emphasize the technical methods or security frames, but also consider the password security under social engineering attacks.

First of all, product managers should balance the security with user experience. For example, many product managers believe that giving different error message when users submit wrong username and password is a way to improve user experience. But in best practice, different error message encourage attackers brute-force the right username and password. Another risk of giving different error message is causing user information disclosure: if the username is user’s email address, by multiple submitting different email address, social engineering attackers can easily get a list of user of your company.

To design an application which can detect unwilling login after user’s privacy disclosure, product managers should always design an “alert” to warn user when login environment changes. An appropriate way to achieve this goal is sending an extra email or SMS message, and giving them an access to “kick out” the attackers.          

Phishing Attack

When we talking about social engineering attacks, we believe that phishing attack is the most common form of social engineering attacks because the cost of phishing attack can be reasonable low and the attack surface of users against phishing attack can be broad.

In this chapter, we provide the recommendations and industrial best practices in countering phishing attack. Since there are many ways to achieve phishing attack, we divide this chapter into many parts to explain phishing attack.

 

Fake Email

Years ago, swindlers used phone calling to achieve fraud cases, in this years, email can help a lot to achieve social engineering attacks. Technically speaking, attackers can forge any email from any domain using their own SMTP server, but we still have some basic methods to identify the email.

For general user, most of email application providers provide their own anti-social engineering technology to help them identify email. If you receive an email from an unsafe domain or forged domain, the application provider would sent you an notification to point out it is an unsafe email. Since every email service provider has their own cloud database to identify the SMTP server, the notification may be always helpful. For general users, there is no need to do additional profound technical detection by themselves, the application providers have already done it for them, so in this guidebook we only talk about how to handle it.

The first rule to handle such emails is just to delete them, and never click the URL in the email, and never download attached files. We will take a more in-depth explanation when we talking about unsafe URL and unsafe attached files.

There is another form of social engineering attack in fake email field is fraud. This kind of fake email may not contain any URL or attached file, but to convince you of something. It is more like a fraud in real world. The most famous fake email fraud case these years is in 2016, an outer attacker send a fake email using his own SMTP server to the CTO of ACCC, which is one of leading digital currency exchange in the U.S.. This email convinced the CTO that the company need 30 bitcoin to test its new bitcoin wallet, and then the CTO just them 30 bitcoin to the bitcoin address the email provided. By using the same methods, the swindler steal 90 bitcoin three times in a row from ACCC.

The way to avoid this kind of social engineering attack is simple, just to double check the information. When you receive an email from your boss, call him or just send him an SMS message to double check.

Unsafe URL

Unsafe URLs are everywhere in the Internet. The purpose of unsafe URL is to steal your private information including account information or cookies. Unsafe URL in social engineering field can be divided into two parts: fraudulent website and URL containing reflected XSS (Cross Site Scripting) attack. (We don’t talk about stored XSS since if an attacker can achieve stored XSS, social engineering attack is not the best they choose to gain illegal profits.)

1. Fraudulent website

Fraudulent website is a kind website that wants to lure you by sharing you “free lunch”. It is a kind of social engineering which is like the fraud in anywhere else. In the vast majority cases, the attackers use secondary domain to camouflage their fraudulent website like google.evil.com to camouflage as google.com. There is no silver bullet to identify the fraudulent website and the real website, so the first standard to identify such a website is to believe there is no free lunch in the world. Thanks to the cloud database and deep learning, many website browser providers have their own database to identify the fraudulent website which can help a lot.

2. URL containing reflected XSS

URLs containing reflected XSS can steal your cookies by lure you to click the URL, it is a more technical social engineering attack combining web application vulnerabilities. For general users, the URL containing reflected XSS can be easily identified by checking whether it contains JavaScript such as “document.cookie” or “window.open”. This kind attack is more dangerous because it is more technical, and users cannot take actions immediately when under such attack. The golden standard to wipe out this kind of attack is on the designer’s side.

For a software engineer or web application designer, the effective way to extinguish this kind attack is avoiding XSS attack. For most cases, simply adding a HTTPONLY flag can prevent user’s cookie from being stolen.

Unsafe Attached File

Unsafe attached files always come with more technical attacks. Thus, if you realize that the attached file is unsafe (which can be detected by the email application providers and antivirus software), it should be the beginning of social engineering attack.

To avoid downloading and opening unsafe attached files, using antivirus software is required. Thanking again to the cloud database, the email application providers can easily check whether the attached file is malicious by checking their signature. We don’t talk about the signature of malicious software since this problem is an application security issue, not social engineering issue.  

Tailgating

In this guidebook, we define that tailgating is an attacker, seeking entry to a restricted area secured by unattended, electronic access control, e.g. by RFID card, simply walks in behind a person who has legitimate access. Following common courtesy, the legitimate person will usually hold the door open for the attacker or the attackers themselves may ask the employee to hold it open for them. The legitimate person may fail to ask for identification for any of several reasons, or may accept an assertion that the attacker has forgotten or lost the appropriate identity token. The attacker may also fake the action ​of​ ​presenting ​an ​​identity ​​token.

The most shocking investigation result is that for most start-ups and small company, the main threat in social engineering field is tailgating. One possible reason is that start-ups may not think outsiders as threats. To avoid tailgating, company should deploy electronic access control and realize that every outsider could be a social engineering attacker. An interesting example is that in 2014, an outsider disguised himself as an air conditioning repairman and steal one of the server from The9 Limited.

There is a simple scenario, when delivery staff or postman walking into your company. At this time, one member of the administration department should follow the delivery staff or postman and guide them finish their work then to leave.
